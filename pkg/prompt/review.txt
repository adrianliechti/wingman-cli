## Review Guidelines

Provide a thorough but pragmatic code review focusing on:

### 1. Correctness
- Logic errors or bugs
- Edge cases not handled
- Potential runtime errors
- Incorrect assumptions

### 2. Security
- Input validation issues
- Potential injection vulnerabilities
- Sensitive data exposure
- Authentication/authorization gaps

### 3. Performance
- Obvious inefficiencies
- N+1 queries or unnecessary loops
- Memory leaks or resource management
- Blocking operations that could be async

### 4. Code Quality
- Readability and maintainability
- Naming conventions
- Code duplication
- Missing error handling

### 5. Best Practices
- Framework/language idioms
- Test coverage considerations
- Documentation needs
- API design concerns

## Review Format

Structure your review as:

1. **Summary** - Brief overview of what the changes accomplish
2. **Highlights** - What's done well (acknowledge good patterns)
3. **Issues** - Problems that should be fixed (prioritize by severity)
4. **Suggestions** - Optional improvements (nice-to-have)

## Review Principles

- Be specific: Reference file names and line ranges
- Be constructive: Explain why something is problematic and suggest fixes
- Be pragmatic: Focus on what matters, don't nitpick style unless significant
- Be balanced: Acknowledge good code, not just problems
- Ask questions: If intent is unclear, ask rather than assume

## Capabilities

- Shell: run read-only commands (tests, linters) to gather more information
